# 天命

天命是一款 AI 驱动的中国古代王朝历史模拟游戏。它把玩家的诏令、朝臣奏报、地方账本、军政财政、民心皇威、派系关系和历史事件放进同一套回合推演里，让每一轮局势变化尽量有因、有账、有后果，而不是只靠随机叙事制造戏剧性。

本仓库是天命的 Web 运行时与 GitHub Pages 仓库。桌面发行版由外层 Electron 壳打包，把本目录作为游戏主体加载；在线展示与静态托管则直接使用本目录。

当前最新公开内容版本：1.2.9.0（2026-05-31）

GitHub Pages：
https://misfit-user.github.io/tianming/

一句话定位：天命不是「AI 写一段古风结果」，而是用 AI 生成叙事和人物反应，再用确定性账本、规则闸门和状态写回约束它。

## 项目定位

天命不是单纯的古风聊天游戏，也不是只给固定剧本套一层 AI 文案。它的目标是做一套可扩展的王朝治理与历史叙事推演框架：

- 玩家通过诏令、军令、廷议、御前问对、地方治理等方式干预国家。
- AI 负责生成奏报、人物反馈、回合叙事、派系行动和局势解释。
- 引擎用确定性账本约束 AI，把财政、民心、皇威、皇权、军队、地方、官制和人物状态落回真实字段。
- 高影响事件有规则闸门，避免无前因的叛乱、政变、黑天鹅和全局崩盘。
- 剧本作者可以用内置编辑器创建其他中国古代朝代的剧本，而不局限于明末或单一历史线。

项目当前以明末天启、绍宋建炎等官方剧本为主要验证场，但底层结构面向多朝代扩展。

## 核心体验循环

天命的一个回合大致围绕六步展开：

1. 读局势：查看奏报、账本、人物、地图、省份、军队、党派、阶层和朝堂反馈。
2. 定方略：决定本回合要处理财政、民心、边防、任免、科举、外交、派系还是地方灾变。
3. 下命令：通过御案诏书、军令、朝议、御前问对、奏疏批复、鸿胪往来等方式落子。
4. 过回合：AI 推演管道综合玩家操作、剧本状态、人物和势力行动，生成下一轮局势。
5. 对真账：财政、民心、皇威、皇权、吏治、军队、地方、人物状态等核心字段尝试写回真实数据。
6. 复盘纠偏：玩家根据新奏报和账本变化继续补救、追责、安抚、扩张或改革。

这套循环的核心不是让玩家「看故事」，而是让玩家能追问：我做了什么，系统是否忠实反映了什么。

## 现在能玩到什么

- 官方剧本：天启七年·九月、绍宋·建炎元年八月等。
- 御案诏书：玩家可拟定政令、军令、外交、经济和其他诏令，新版「御案」界面支持议题纳入、分类拟旨、有司润色与实时预测。
- 回合推演：过回合后，系统会综合玩家操作、剧本状态、人物势力、地方账本和 AI 反馈生成新局势。
- 朝会与问对：常朝、廷议、御前、问对、奏疏、鸿胪外交等模块共同构成朝廷交互面。
- 人物与势力：人物有官职、派系、关系、行动记录与 AI 生成的奏议/反馈；势力会参与博弈而不是只做静态标签。
- 财政与地方：央地财政、实征率、浊度、税粮、地方民心、腐败、人口与资源会参与推演。
- 科举与士林：恩科、童子、武举、改革派、门生网络、言官清议等科举相关系统正在持续扩展。
- 地图与区划：地图编辑器和行政区划数据已接入，支持省府州县、地形、资源、人口、民心、腐败、财政等多层信息。
- 剧本编辑器：内置编辑器支持人物、官制、势力、行政区划、财政、军政、事件材料和 AI 辅助生成。
- 创意工坊基础：已有工坊包格式、在线目录、资源引用和更新链路设计，玩家发布与安装自制剧本仍在建设中。
- 热更新：已有 hot update 包格式与构建脚本，支持不等安装包大版本也能分发 Web 运行时与官方剧本改动。

## 为什么不是普通 AI 聊天

普通 AI 聊天游戏往往是「玩家输入一句话，AI 接一段故事」。天命的目标更重：让 AI 只负责它擅长的叙事、解释、人物语气和局势发散，把会影响玩法的核心状态交给账本和规则层兜住。

天命当前采用的是混合管道：

- AI 负责把复杂朝政写成人能读懂的奏报、辩论、人物反应和局势解释。
- prompt、schema、validator 会要求 AI 输出结构化字段，而不只是散文。
- change applier 会把 AI 输出拆成财政、人物、军队、地方、叙事等变更。
- 财政、民心、皇威、吏治、腐败、军队、科举、势力等模块再把这些变更落回各自账本。
- 高风险事件还有相称性闸门：叛乱、政变、劫狱、抗命、黑天鹅必须尽量有前因、范围和代价链。

这意味着天命追求的是「叙事可读 + 状态可查 + 结果可追责」。它还不是全确定性模拟器，也不承诺 AI 永远不跑偏；但工程方向是把跑偏处逐步收束进真账。

- 新御案诏书润色改为浮层卡，不再挤掉五栏正文；采用润色稿会同步新御案草稿账本。

- 民变不再在民心安定省份凭空坐实，新生叛军规模按省人口封顶。
- 文风难度统一为叙事、标准、硬核三档，并真正接入推演节奏。
- 黑天鹅与强行转折按难度收束，叙事档会更尊重玩家治绩。
- 皇权或皇威正盛时，AI 不能凭空坐实政变/弑君成功。
- 各省民心并回同一本真账，民情图与省份面板读取真实当前值。

## AI 推演与账本

AI 推演是天命的核心，但不是唯一核心。它由「回合管道、提示词、结构化输出、校验、变更应用、账本写回」共同组成。

主要代码落点：

- tm-endturn-core.js 负责回合主入口和管道组织。
- tm-endturn-pipeline-steps.js 负责准备阶段、AI 阶段、系统结算阶段等分段流程。
- tm-endturn-ai.js 负责主要 AI 调用、结构化回合结果和失败兜底。
- tm-prompt-composer.js 负责把当前局势、玩家操作、账本、规则和输出要求组织成提示词。
- tm-ai-output-validator.js 负责对 AI 输出做结构检查和风险提示。
- tm-ai-change-applier.js、tm-ai-change-army.js、tm-ai-change-narrative.js 等负责把 AI 输出落成状态变化。

账本侧不是一个单文件，而是一组专题引擎：

- tm-fiscal-engine.js 管国库、财政、实征、央地收支等。
- tm-authority-engines.js 管民心、皇威、皇权等权威指标。
- tm-corruption-engine.js 管地方腐败、浊度和治理效果。
- tm-central-local-engine.js 管央地关系和地方承接。
- tm-huji-engine.js、tm-economy-engine.js 等补充人口、户籍、经济条件。

这套设计的关键点是：AI 可以提出「陕西局势恶化」「某派系不满」「某地粮价波动」，但真正影响玩家后续判断的内容，最终要尽量落到可查字段和可复盘记录里。

## 御案、朝会、奏疏、问对、科举

朝廷交互是玩家影响世界的主入口。它不是单一按钮，而是一组互相接线的制度界面。

御案与诏书：

- phase8-formal-drafts.js、phase8-formal-*.js 负责新版御案界面、草稿、记录、地图、右栏、顶栏等正式界面层。
- tm-edict-parser.js 负责识别玩家诏令意图。
- tm-endturn-edict.js、tm-edict-lifecycle.js 负责把诏令带入回合推演与生命周期。
- tm-hongyan-office.js 承接有司润色、鸿胪往来和部分文本处理。

朝会与议政：

- tm-chaoyi.js 是朝议通用入口。
- tm-chaoyi-changchao.js 管常朝。
- tm-chaoyi-tinyi.js 管廷议。
- tm-chaoyi-yuqian.js 管御前。
- 朝会不是纯文本表演，目标是把官员立场、派系目标、人物状态、议题结果和后续政策链路接起来。

奏疏与问对：

- tm-memorials.js 管百官奏疏、批复、留中、发廷议等。
- tm-wendui.js 管御前问对。
- tm-hongyan-office.js 管鸿胪、外交、润色和部分文书交互。

科举与士林：

- tm-keju.js 是科举系统主入口之一。
- tm-keju-runtime.js、tm-keju-paradigm-panel.js、tm-keju-reform-apply.js 等承接科举改革范式、运行时状态和改革落地。
- tm-keju-enke.js、tm-keju-wuju.js、tm-keju-tongzi.js 等扩展恩科、武举、童子等专题。
- 科举系统正在从「考试事件」扩展为士林、言官、门生、改革派和政治资源网络。

这些模块的共同目标是：玩家不是只在对话框里说一句「改革科举」，而是要经历提案、廷议/科议、表决、推行、阻力、记录和后续影响。

## 地图、剧本编辑器、创意工坊

地图系统、剧本编辑器和创意工坊决定了天命能不能从一个官方剧本扩展成多朝代、多作者、多玩法的框架。

地图与区划：

- map-editor.html、map-editor-*.js 提供地图编辑能力。
- data/maps/ 保存官方地图、行政层级、地块和归属数据。
- 当前地图填色按区域归属和势力关系上色；是否变色取决于推演是否把「易主」落成结构化地块归属，而不是只在奏报里写一句捷报。
- 当前地图粒度更接近大区、都司、布政使司等层级；单城克复未必等于整块区域立刻变色。

剧本编辑器：

- editor.html 与 editor-*.js 是剧本编辑器主体。
- 支持朝代、年号、人物、官制、势力、行政区划、财政、军政、事件材料、历史补充数据等。
- editor-fullgen.js、editor-schema-adapter.js 等用于 AI 辅助生成、结构适配和数据补全。
- 剧本编辑器的长期目标是让作者能创建一个可推演的朝代，而不是只改几段开局文本。

创意工坊：

- tm-content-manager.js 承接内容包管理和运行时加载方向。
- docs/workshop-pack-format.md 描述工坊包格式。
- docs/online-distribution-server.md 描述在线分发服务思路。
- 当前已有包格式、资源引用、在线目录和更新链路设计；完整玩家发布、安装、更新、审核和版本兼容仍在建设中。

README 不把工坊写成已经完整上线的平台。准确口径是：底层格式和分发设计已存在，产品化闭环还在做。

## 发布、热更、Godot

天命现在有三条并行但用途不同的工程线：

1. Web 运行时

- 本仓库根目录就是 Web 运行时。
- index.html 按顺序加载大量原生 JS 文件。
- GitHub Pages 用这个目录做静态展示。

2. Electron 桌面本体

- 完整桌面工程在外层 tianming/ 目录。
- 桌面本体负责窗口、文件系统、安装包和本地运行环境。
- 公开安装包版本可能低于 Web 内容版本，这就是 1.2.8.9 热更门槛问题的背景。

3. Godot 迁移实验

- godot/ 是 Godot 4.6 迁移工作区。
- 已有标题页、运行时壳、面板组件、存读档、军队、地图、事件、关系、治理等大量验证脚本。
- 当前正式发行仍以 Electron + Vanilla JS 为准；Godot 不是已替换上线的主版本。

热更新链路：

- tools/build-hot-update-package.js 构建 hot update 包。
- release-hot/hot-latest.json 指向最新热更。
- release-hot/manifests/ 保存版本 manifest。
- changelog.json 是公开邸报来源，最新条目在数组最前。

发布时要同时看四件事：GitHub 代码、changelog 邸报、hot-latest 指针、manifest 的 minAppVersion。只推其中一项都可能造成「版本显示新了，但功能没到」。

## 快速开始

作为静态 Web 运行时查看：

1. 克隆本仓库。
2. 在仓库根目录启动任意静态文件服务器。
3. 打开 index.html。

示例：

python -m http.server 8080

然后访问：

http://localhost:8080/

作为桌面版开发运行：

本仓库是 web 运行时。完整桌面工程在本地外层目录 tianming/ 中，通过 Electron 加载 web/。在完整工程根目录执行：

npm install
npm start

Windows 本地开发也可以使用外层目录的 启动天命.bat 或 dev-start.bat。

## 版本与热更新

天命的桌面本体和 Web 运行时不是同一个版本号口径：

- 桌面公开安装包可能仍显示较低的本体版本，例如 1.2.2。
- 游戏主体功能主要由 Web 运行时承载，可通过热更新滚动到 1.2.9.0。
- hot-latest.json 指向最新热更包；manifest 中的 minAppVersion 决定当前本体能否吃到该热更。
- 如果玩家看到“无法完成热更”或“本体版本过低”，优先检查热更门槛，而不是直接要求玩家重装完整安装包。
- 1.2.8.9 已把热更最低本体版本降回 1.2.1.0，用来兼容当前公开本体 1.2.2。
- 1.2.8.10 之后的累计热更包补齐旧本体缺失的前端主链路，避免「版本新、UI 旧」。

因此，README 中的“最新公开内容版本”指当前公开可取得的游戏内容版本，不等同于 Electron 外壳本体版本。

## 当前限制与路线图

当前限制：

- AI 推演仍可能跑偏，所以系统持续把高风险叙事收回到账本、闸门和校验层。
- 地图变色依赖结构化区域归属变更；如果推演只写文字捷报而没有改归属字段，地图不会自动变色。
- 省份、阶层、党派、军队等面板仍在逐字段做活，部分字段可能先用于展示或填色，后续再接入推演。
- 创意工坊已有格式与分发设计，但还不是完整的 Steam Workshop 式产品化平台。
- Godot 工程是迁移实验，不代表当前正式发行已经从 Electron 切走。
- Web 运行时仍是大量原生 JS 顺序加载，改全局字段和加载顺序要非常谨慎。

近期路线图：

- 继续把科举改革、阶层、党派、军队、地方省份字段接入真实推演闭环。
- 补强地图易主、区域控制和战果落账，避免「打赢了，地图不认」。
- 完善创意工坊的发布、安装、更新、校验和版本兼容。
- 推进完整安装包与热更新版本口径统一，减少玩家对本体版本和内容版本的误解。
- 继续整理 README、系统文档和开发报告，让外部贡献者能看懂项目结构。
- 稳步推进 Godot 迁移，但在可验证前不替代当前 Electron 主发行。

## 剧本编辑器

剧本编辑器是天命的重要组成部分，不是附属调试页。它的目标是让剧本作者能创建和维护一个朝代的基础世界：

- 朝代、年号、开局时间与初始皇命。
- 皇帝、朝臣、地方官、武将、后妃、士人、势力人物。
- 官制树、部门、官职、任命关系与权责。
- 行政区划、地方财政、人口、资源、地形、民心与腐败。
- 军队、边镇、装备、军饷、士气、训练与主帅。
- 历史事件、危机材料、人物传记、势力目标与叙事素材。
- AI 辅助生成、校验、修复和多轮补全。

当前编辑器相关代码主要在 editor.html 与 editor-*.js。

## 设计原则

坏事可以发生，但必须讲得通。

天命允许灾荒、党争、兵变、财政崩坏、边患失控和权威受损。问题不在于坏结局本身，而在于坏结局必须有明确前因、范围、代价链和可处理入口。

因此项目持续强化三层约束：

- 账本约束：关键指标要落到真实字段，不允许只写叙事不改状态。
- 相称性约束：叛乱、政变、天灾、抗命、劫狱等高影响事件必须与民心、皇威、看押强度、人物忠诚、地方状态相称。
- 难度约束：叙事、标准、硬核三档应真正影响节奏，而不是只换名字。

玩家最有价值的反馈不是「我不爽」，而是「我做了什么，系统没有忠实反映什么」。

## 开发注意

天命 Web 运行时是零编译、纯原生 JS、无模块系统的项目。脚本通过 index.html 顺序加载，大量能力挂在 window 全局对象上。

改动时请注意：

- 新增脚本必须登记进 index.html，并放在依赖之后。
- 不要随意调整加载顺序。
- 改全局函数或字段前，先全仓搜索调用点。
- GM 和 P 是核心状态对象，字段改名会影响旧存档兼容。
- 改完 JS 至少跑 node --check。
- 涉及回合推演、存档、人物移动、财政民心、AI 输出解析的改动要配 smoke 测试。

更详细的协作守则见外层工程 DEV-GUIDE.md。

- 游戏版本或 changelog 日期。
