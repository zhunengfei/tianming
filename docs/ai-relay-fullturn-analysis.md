# 全回合演绎脑实跑分析（天启七年·有操作回合·T1→T2）

> 数据来源：2026-05-29 经本地中转、由 Claude 充当演绎脑、零外部 API 跑通的一个**有操作回合**（召见 4 人 + 陕灾三人各两轮廷议 + 救陕/恩科融合大诏 + 朔朝 9 人朝议），endTurn 管道完整走完、游戏推进到 T2。中转 transcript：`queue/transcript.jsonl`（58 调用）。

## 一、输入体量总览（硬数据）

| 指标 | 值 |

## 二、最大问题（按严重度排序，全部有本回合实证）

### ❶ 静态 sysP 重复 18 次 ≈ 全回合 69% 的输入（**真正的头号成本**）
那份 54.8K 字的系统提示——剧本总述、22 势力、24+ NPC 名单、地名名单、官制分工、NPC 弧线、隐藏议程、危机分岔、后宫/门阀规则——**几乎全是静态的**，却在一回合内随 sc0/sc1/sc1b/c/d/sc15/16/17/18/sc05/NPC认知/sc27/sc28/sc25c 等 **18 个调用各重发一遍**，合计近 **百万字 / ≈300K tokens**，远超 sc1 单次 tp1（114K 字）的体量。
- **三刀 sprint 只砍了 sc1 的 user(tp1)**，没碰这块——**真正的大头在这里**。
- 第三方中转拿不到 prompt 缓存（`_maybeCacheSys` 只对 anthropic/openai 原生生效），所以这 18 份每份都是全额计费。
- **这是最该优化的点**：把静态 sysP 与每回合变动部分分离，turn 内对 sc1-family 复用同一份缓存/摘要，可省全回合约 6-7 成输入。

### ❷ sc1 单次 170K 字 = weak-model 的双重失败点
sc1 既是最贵（51K tokens 输入）、又要求最复杂的输出（AIScenarioResponse：zhengwen + events + npc_actions + edict_feedback + char_updates + office_assignments + personnel_changes + fiscal_adjustments + economic_advice + tableEdit + 5 条硬要求，且"第一字必须 `{`、最后一字必须 `}`、否则整回合降级"）。
- 弱模型在 170K 上下文里既容易丢失中段指令，又容易吐出非法 JSON → 整条 sc1b/c/d/sc2 连锁降级。
- 实测我（强模型）尚需逐字校验；换 gpt-4o-mini 级别大概率崩。

### ❸ QC/安全网调用过多（4 个），且暴露 schema 不一致
一回合里有 **4 个纯质检调用**：sc_audit 一致性自审（050，tool_calls 回填）、跨 sub-call 一致性审核（058）、JSON 修复（060）、叙事质量审查（062）。
- 它们确实有用——**050 真抓出我 sc1 的 `fiscal_adjustments` 没被解析进去**，我用 tool_calls 回填了财政/天灾/科举/工程；说明 sc1 主输出常漏落地。
- 但 **060（JSON 修复）暴露了一个真问题**：057 调用的 prompt 大讲 `edict_lifecycle_update`，可解析器实际要的是 `houren_xishuo/zhengwen/new_activities`——**prompt 写的 schema ≠ 解析器期待的 schema**，靠修复网兜回来，白烧一个调用。这类"提示与解析器不同步"值得排查。

### ❹ 常朝逐人调用：N 个在场官员 = N 次调用（同一议题）
朔朝"杨所修劾崔呈秀夺情"一个议题，**9 个朝臣各一次调用**（李养正/倪元璐/许显纯/周延儒/薛凤翔/冯铨/李标/刘诏/薛贞），各带 mode（soften/rebut/second）+ 必含词 + 字数硬下限。
- 在场 20 官 → 可达 20 次调用 / 单议题。
- 且 mode 必含约束让发言**趋于模板化**（"附议…然…伏乞圣裁"），多样性被规则压制。
- 可考虑：一次调用批量生成 N 人表态、或按重要度采样 3-5 人实答其余降级。

### ❺ weak-model robustness：硬约束密集，乱码/英文泄漏频发
大量调用有"必含某词/字数硬下限/纯 JSON/无 markdown/无 ASCII 引号"等硬约束。本回合我（及子 agent）就观察到模型**自发吐出乱码/英文残留**：016 的 `trajectoryN�xt10Turns`（U+FFFD）、林丹汗 `north面`、魏忠贤 `年迈multi病侵`、薛贞 `体statute`。这些都是**模型层瑕疵**而非游戏问题，但游戏若无逐字校验就会把脏数据写进存档（静默丢字段，最危险）。**中转/落地层应按 schema 校验 + 探测替换字符后再交付**（我已在每个 builder 加了这层）。

## 四、三刀最终复盘（原始 sprint 目标的结论）

| 刀 | 结论（全回合实证） |
|---|---|
| **第①刀 问对摘要化** | ✅ 在线工作。sc1 内问对块=摘要 631 字，全文（1015+字）转 sc1q。**但字节收益小（净省约 400 字 / <1% of tp1）**——因为游戏对存储的问对本就截断，且 tp1 主体是静态势力/史料。其真正价值在**避免问对全文双重注入 + 不丢信息（语气/承诺经 sc1q 结构化回灌）**，非省 token。 |
| **第②刀 军事砍除** | ✅ 确认成功。sc1 已无兵种/年饷；军事明细确**完整出现在 sc18(054)** 的 prompt 里——relocation 落地。 |
| **第③刀 静态人物背景外移** | ❌ 未做。**而这正指向真正的头号问题**（见 §二❶）：静态 sysP×18 才是全回合 69% 的输入。 |
| 跨存档对比 | sc1 tp1：baseline 116676 / run1 空操作 109179 / 本次有操作 114794。**有操作却比 baseline 还低 1.6%**——说明三刀把这一整车操作内容"吸收"住了，重操作回合不膨胀（这是三刀在有操作回合的真实价值）。 |

## 五、亟需完善（优先级建议）

1. **【最高·省 ~6-7 成输入】turn 内 sysP 去重/缓存**：把 54.8K 静态 sysP 与每回合变动段分离；sc1-family 18 个调用复用同一份（或对走原生 API 的渠道启用 prompt cache、对中转渠道做 turn 级 system 复用）。
2. **【高】sc1 减负 + 防降级**：继续把 sc1 的"可专项化"内容外移到 sc15/16/17/18（军事已移）；对 sc1 输出加服务端 schema 校验 + 自动补全，减少对 4 个 QC 调用的依赖。
3. **【中】QC 调用合并**：sc_audit / 一致性审核 / JSON修复 / 叙事审查 可考虑合并为 1-2 个；并排查 057↔解析器的 schema 不同步（白烧修复调用）。
4. **【中】常朝批量化**：单议题多人表态改为一次批量生成、或按重要度采样实答，降调用数、解模板化。
5. **【底层】落地层按 schema 校验 + U+FFFD/英文残留探测**：防弱模型脏输出静默写库（中转侧或游戏侧都应做）。
