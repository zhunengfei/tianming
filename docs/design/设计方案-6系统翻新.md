## 0 · 翻新背景与核心问题

### 0.1 现状诊断

天命的 6 个引擎层子系统（人物·阶层·党派·官制·势力·军事）目前的整体形态是 **"编辑器面丰富·运行时面空转"**：

- 剧本编辑器里能填的字段非常完整（如阶层 17 字段·党派 23 字段·势力六维凝聚力 + 军事结构 + 经济结构 + 继承）
- 但游戏内运行时引擎不读这些数字·或读了不动——"东林党·影响力 80"30 回合还是 80
- AI 在叙事中提及但不写回·形成"看见的世界"和"机制的世界"两层割裂

### 0.2 翻新目标

不是从零做 6 个新系统·而是**给现有系统补"动力学"**·把静态字段转成动态指标·让玩家诏令真的能改这些数字·让 AI 推演产出可校验的结构化结果·而不只是叙事文本。

### 0.3 不做什么

- 不重做编辑器（编辑器已经够丰富·只在缺字段时扩展）
- 不重做 AI prompt 框架（已有 sc1/sc1c/sc23 等子调用结构·只在 schema 上加固）
- 不重做存档格式（用迁移框架兼容旧档·旧字段保留作 display）
- 不动人际关系（`tm-relations.js` 25 关系标签 + 22 互动类型 + 23 势力互动已是天命资产）

## 1 · 核心设计原则（贯穿 7 阶段）

### 1.1 双层调控原则【最重要】

天命的本质是 **AI 历史演绎**·不是引擎掷骰子。任何"自动 tick → 出结果"的 hardcoded 路径·必须改成 **"引擎算 baseline + AI 必读 acknowledge + 极端 ceiling 兜底"** 三层结构。

**三层职责**：
- **引擎层**·算 baseline + 写信号·公式产出"该发生什么"的初始数值/状态
- **AI 层**·必须读到信号 + 必须 acknowledge·可调整 ±30% 或推翻但要给理由
- **Hard ceiling**·极端情况强制兜底·防止 AI 软懒永不触发或永远跑离 baseline

**唯一允许 hardcoded 的两类场景**：
1. **数据完整性约束**：编制 5 名·满了不能再任命
2. **物理/经济常识**：将领死了部队没有指挥；morale<10 直接 routed（非政治判断）

其它涉及"是否触发""是否升级""谁站谁那边""如何反应"的·全交 AI。

### 1.2 描述化原则

UI 一律用状态文字·不显数值。沿用现有 `partyStrife` 5 档（朝堂清明 / 朝局稳健 / 党争寻常 / 党争激烈 / 党祸滔天）的范式·扩展到所有新引擎指标。

| 原 | 改 |
|---|---|
| "士绅满意度 30/100" | "士绅多有怨言" |
| "东林党 cohesion 75" | "东林党气宇尚整" |
| "藩镇 controlLevel 80" | "藩镇近乎自主" |

### 1.3 剧本完全自包含原则（Q1=B）

引擎不藏默认值·所有官品阶梯/兵制/战术/铨选/加衔等常量必须由剧本在 `scriptData.engineConstants.*` 显式声明。引擎读不到默认·按"未声明"处理。

**Bootstrap UX**：剧本编辑器加 4 朝模板按钮（汉/唐/明/清）+ 通用古制兜底·点击后把模板填进当前剧本（属于剧本不属于引擎）。

### 1.4 关系图渐进引入（Q2=C）

`tm-relations.js` 已有的 25 关系标签 + 5 维关系（affinity/trust/respect/fear/hostility）+ history[] 不动。新增"实体间引用"（角色 → 党派 / 角色 → 势力 / 角色 → 官位）才用 typed ref·不强制全字段 ref 化。

### 1.5 字段命名 + 兼容窗口（Q3=A·Q4=温和）

- 后缀 Ref 风格：`ch.partyRef` / `ch.factionRef` / `ch.officeRef`
- 旧字段保留作 display·写入只走新字段·读处可任选（参考 `zi/courtesyName` 同步先例）

### 1.6 朝代覆盖

**核心 4 朝**：汉 / 唐 / 明 / 清（覆盖大多数玩家创作）
**通用古制兜底**：宋 / 元 / 五代 / 辽金西夏 / 南北朝 / 秦隋等都走"通用古制"模板·剧本作者可在通用基础上自定义。

## 4 · 阶段 1·阶层引擎

### 4.1 现状

`scriptData.classes[]` 编辑器丰富（满意度/影响力/unrestLevels 4 级阈值/economicIndicators 三项/representativeNpcs/leaders/supportingParties/regionalVariants/internalFaction）·但运行时永远是初始值。`tm-phase-f3-depth.js` 只动 demographic 人头数（byClass.peasant_self/peasant_tenant）·不动命名阶层情绪。两套 class 数据并存且互不读。

### 4.2 翻新

**核心**·给 satisfaction/influence/unrestLevels 接动力学·让诏令真能改这些数字。

#### 4.2.1 诏令 tag 系统

- 新增 7 类 tag：赋税/徭役/特权/宗教/军事征发/法律/外贸
- 编辑器·剧本作者写诏令模板时勾选
- 玩家临时下令·引擎用关键词识别（"加征"→赋税·"征发"→徭役）
- 识别不出走 AI 兜底
- tag × 阶层 baseline delta 矩阵·剧本可在 engineConstants 自定义

#### 4.2.2 阶层情绪驱动·双层调控

- **引擎层**·诏令 tag 命中 → 算每阶层 satisfaction baseline delta（如赋税 -5·士绅 -2 / 自耕农 -8 / 佃农 -10）
- **AI 层**·sc1c 看到 baseline·**可在 ±30% 内调整**·调整必须给理由（"江南士绅尤其不满·额外 -3"）·总幅度 clamp 在 baseline ± 30% 内
- **Hard ceiling**·调整超 ±30% schema 拒绝·提示 AI 重选

#### 4.2.3 unrestLevels 跌破·双层调控

- **引擎层**·跌破 grievance/petition/strike/revolt 各阈值 → 写 `GM.minxin.alerts[]`·级别标注
- **AI 层**·sc1c 必读 alerts·必须在 class_response 中 acknowledge 每条 alert（触发事件 / 不触发但解释 / 部分触发）·**不允许沉默忽略**
- **Hard ceiling**·跌破 95 持续 3 回合·strong prompt"该阶层临界·必须出叙事"

#### 4.2.4 民变 brewing → uprising 升级·双层调控

- **引擎层**·brewing 阶段开始时启 timer·每回合写"已 brewing N 回合"
- **AI 层**·N 回合后强制三选一：(a) 升级 uprising / (b) 解除 brewing 但说原因 / (c) 维持但说理由·**不允许沉默**
- **Hard ceiling**·brewing 持续 12 回合·引擎给"必须升级"信号·AI 还要选 a/b 但拒绝 c

#### 4.2.5 桥接器

- `scriptData.classes[].size` ↔ `GM.population.byClass.*.mouths` 双向同步·改一边按比例算另一边

#### 4.2.6 阶层面板

- 显示状态文字（"民怨渐生""士绅有怨言"）·**不显数值**
- 点进去看影响来源·列最近 5 条诏令/事件
- 沿用 partyStrife 描述化风格

#### 4.2.7 阶层 ↔ 党派联动

留待 phase 2 一并做（避免单向耦合写出来过 1-2 周又要回改）。

### 4.3 工作量

**3-4 天**·依赖阶段 0。

## 12 · 风险与未决问题

### 12.1 已知风险

1. **AI schema 强制 acknowledge 可能让 prompt 过长**·token 成本上升
   - **缓解**：sc1c/sc23 的 schema 字段做 conditional·只在有 alert/ceiling 时才注入
2. **双层调控的 ±30% 上限**·需要测试调整·初期可能太宽或太窄
   - **缓解**：上限做剧本可调·engineConstants.aiAdjustClamp = 0.3
3. **4 朝模板考据·明清相对清晰·汉唐有多次改制**
   - **缓解**：每朝选一个标志性时间点（汉=武帝时·唐=开元后·明=永乐后·清=雍正后）·标注模板基于哪个时点
4. **存档迁移幂等性**·跨版本升级时可能多次 hook 调用
   - **缓解**：每个 migration hook 写入版本号·重跑时 skip

### 12.2 未决问题

1. **阶段 1 的 tag 矩阵 49 格 × 4 朝 = 196 格 baseline delta**·是否做工具批量生成？
   - **建议**：先手填明朝（参考工程量最熟悉）·再用 AI 推演辅助生成其它 3 朝·人工校对
2. **阶段 6 的人物 ref 化是否要给现有剧本作者升级工具**？
   - **建议**：编辑器加"诊断模式"·扫描剧本中所有 ch.party 字符串·提示哪些找不到对应 partyRef·让作者补正
3. **战斗 verdict schema 的 occupiedCityIds 写回路径**·若 AI 误填不存在的城市 ID 怎么办？
   - **建议**：BattleEngine 写回前校验 GM.cities 存在·不存在则 silent fail + 写错误日志·不阻断游戏
4. **双层调控的 baseline 公式·是否所有都在 engineConstants 暴露给剧本作者**？
   - **建议**：核心几个（阶层 satisfaction baseline·党派 influence·军事 morale 减幅）暴露·其它内部公式不暴露
