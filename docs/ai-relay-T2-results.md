## 三、bug 样本（游戏的锅·未改·留作查 bug）
> owner 指示：出错先判游戏锅还是我锅，一律按游戏 schema 生成、不改游戏，方便后续查 bug。以下均判定为**游戏侧**问题。

- **A. sc1 的 `fiscal_adjustments` 字段不被解析进账本**（稳定复现 T1）。我 sc1(006) 明确输出 `fiscal_adjustments:[{关宁军饷 -200000}]`、JSON 合法，但 sc_audit(010) 体检显示 `fiscal_adjustments: []`。游戏的 sc1 解析器没提取该字段。我按游戏提供的 tool_calls 途径（`record_fiscal_adjustments`）补录了辽饷 20 万——这正是游戏自己的兜底机制，说明引擎"知道"该字段常漏、却没在 sc1 主解析里接好。
- **B. sc27 prompt 框架 vs 输出契约错位**（同 T1 057）。016 prompt 大段讲"诏令 9 阶段生命周期·颁布≠见效"，但最终输出契约只要 `{houren_xishuo, new_activities}`——框架与字段名不对应。我严格按输出契约填，未臆改。
- **C. T3 候选事件池未随诏令更新**（新发现）。T3 开局事件池仍写"杨所修疏劾崔呈秀夺情起复"，但崔呈秀在 T2 已被玩家诏令勒令回籍——事件池是预计算的、没消费玩家已解决的事件。

## 四、踩坑教训（已写入 RESUME 与 memory）
- 必须先注入"去 fetch 超时"补丁（poll-and-secure.js 已含），否则游戏 ~45s 超时触发重试风暴、队列变脏。
- reload 页面会卡死 cdp-eval；重启游戏用"全新启动 electron"。
- Write 工具偶发 ENOENT（疑杀软扫 answer 目录）；node `fs.writeFileSync` / bash `echo >` 兜底可靠。
- config 重定向是运行时注入·每次重启/reload 丢·必须重注入；poll-and-secure 把重定向+去超时合一防竞态。
