## 子调用 1: SC1 (结构化数据·主推演)

位置: L1167-2580 | 最大单一函数 | 关键通路

### 1. 目的与边界分析

核心目的: 解析玩家诏令、NPC奏疏、事件触发，生成本回合结构化账本
- 60+顶级字段、预算占比 45%

职责范围:
- 生成账本基础 (turn_summary, shizhengji_basis)
- 角色/势力/阶层状态增删
- 诏令追踪与反馈
- 财政/军事/外交/人事变化

与其他SC1x的边界:
- SC1d (L2583-2648): SC1 输出 → 改写为实录/时政记文言文形式(只改写,不新增数据)
- SC1b (L2651-2810): 独立注入·文事/鸿雁/NPC互动(不依赖SC1输出)
- SC1c (L2812-3223): 独立注入·势力外交/NPC阴谋(不依赖SC1输出)

边界清晰度: 中等 (3/10)
- OK: SC1b/SC1c与SC1无交集字段(并行执行安全)
- 问题: SC1d严度依赖SC1输出(若SC1失败则SC1d无法成文)
- 问题: 硬约束注入占600字,压缩度80%,规则重复密集

### 2. 设计质量评估

Schema 膨胀问题 (致命):
- 60+ 顶级字段,代表表包括:
  * 人事 (char/personnel/office): 8+ 字段,权重 40%
  * 势力/党派 (faction/party): 12+ 字段,权重 25%
  * 财政/资源: 6+ 字段,权重 15%
  * 诏令/生命周期: 5+ 字段,权重 12%
  * 事件/记忆/表: 8+ 字段,权重 8%

问题:
- 字段间明显嵌套冗余 (如char_updates + office_changes都能表达任免)
- Prompt 膨胀: 单行 tp1 构建已含 25K tokens
- 验证策略混乱: 某字段既能在char_updates.updates.任意字段,又能在char_updates.careerEvent
- 必填vs可选界限不清: 大多字段(50+)标可选,AI创意生成导致格式破损增加

Schema 清晰度评分: 3/10

Prompt 指令化程度 (过度):
- L1172-1193 硬约束分析:
  * 5条硬约束,各占60-80字,共380字
  * 问题: ① 金额一致性/死亡禁动/数据与叙事重复出现在sc_audit中
  *       ② 忠诚语义/角色归属可抽为全局system prompt
  *       ③ 铁律式规则不利于JSON生成

冗余规则数: 2-3条可删,成本100字prompt,收益减少合规焦虑

指令密度评分: 4/10 (建议<=2/10)

### 3. Token 经济分析

Input Token 预算 (假设gpt-4o):
- System prompt: 4000-5000字 = 1500-1800 tokens (5%)
- 上回合记忆固化: 1500-2000字 = 400-600 tokens (1.5%)
- 状态板: 800-1200字 = 200-300 tokens (0.8%)
- 硬约束+世界快照+12表: 3000-4000字 = 800-1000 tokens (3%)
- 核心prompt (tp1后): 18000-22000字 = 4500-5500 tokens (17%)
- 玩家诏令/奏疏历史: 2000-3000字 = 500-800 tokens (2%)
- 总计: ~30000字 = ~8500-9500 tokens (30%)

Output Token 预算:
- 目标: 7000 tokens
- 实际常被截断到4000-5000
- 实际生成: 平均5000-6000 tokens(占输出总预算50-60%)

Token ROI 分析:
- 单回合成本: 9500 input + 5500 output = 15000 tokens
- 按gpt-4o: $0.005/1K input + $0.015/1K output ≈ $0.13/回合
- 年度成本(100回合): $13

Token 经济评分: 5/10 (改进空间大·可通过拆分降40-50%)

### 4. 质量天花板分析

问题: 60+字段真的都需要吗?

字段优先级评估:
- 必含: turn_summary, char_updates, resource_changes, fiscal_adjustments 等
  使用率100%, 下游sc2/sc16/sc17依赖
- 高频: edict_feedback, npc_actions, personnel_changes 等
  使用率80%+, 诏令生命周期+任免追踪
- 中频: faction_changes, party_changes, goal_updates 等
  使用率50-70%, 势力党派/NPC轨迹变化
- 低频/可选: office_spawn, party_agenda_shift, hidden_moves 等
  使用率<20%, 创意/扩展·不影响核心
- 冗余: class_updates等 (若已有阶层系统)
  使用率~5%, 可归纳到faction_changes

裁剪到30字段的可行性: 70%可行

必含(10字): turn_summary, shizhengji_basis, char_updates, personnel_changes,
  office_changes, edict_feedback, fiscal_adjustments, resource_changes,
  faction_changes, events

高频(10字): npc_actions, affinity_changes, goal_updates, party_changes,
  faction_events, faction_relation_changes, army_changes, cultural_works,
  npc_letters, npc_interactions

改进方案:
- 分层schema: 必含(strict) / 高频(expected) / 可选(nice-to-have)三层
- AI创意引导: 改为按优先级填·缺若干字段不扣分

质量天花板评分: 6/10 (实际可达8/10若裁剪到30字段)

### 5. 与其他SC1x的冗余性分析

SC1与SC1b冗余:
- SC1包含: cultural_works, npc_letters, npc_interactions
- SC1b也包含: 完全相同
- 方案: SC1删去这三字段·转由SC1b专项生成
- 收益: SC1 prompt缩20%, SC1b独立职责更清

SC1与SC1c冗余:
- SC1包含: faction_events, faction_changes, npc_schemes
- SC1c也包含: 类似字段
- 方案: SC1删掉势力互动·交给SC1c专项
- 收益: SC1 prompt缩25%

SC1与SC1d依赖:
- SC1输出 → SC1d输入
- 设计问题: SC1d不能完全独立(必须等SC1完成)
- 若SC1失败则SC1d无法救援
- 推荐改进: 让SC1d同时接收player_status/edict_feedback/event源数据,可自行成文

冗余度评分: 7/10 (严重·可通过重组化解50-60%)

### 6. 并行化潜力

- SC1: 串行输出p1
- SC1d/SC1b/SC1c: 按Promise.all并行等待p1
- 3并行子调用间零交集

SC1本身能否拆分?

- SC1-人事 (10字): char_updates, personnel_changes, office等
- SC1-财政 (8字): fiscal_adjustments, resource_changes等
- SC1-势力 (12字): faction*, party*, vassal_changes等

- OK: 并行度4·时间收益显著
- 问题: 诏令反馈涉及跨族字段(需在汇总层做一致性检查)

收益评估:

- 质量: +15%(每个子任务专注度提升)

SC1并行化评分: 7/10 (强烈推荐·ROI明显)

### 7. 失败链分析

SC1失败的致命性:
```
SC1失败(0%成功率)
  → p1 = null
  → L2500 rescue触发(_trySc1Rescue)
     - 成功: 用轻量结构化账本替补(partial)
     - 失败: p1仍null → toast提示 + continue
  → GM._turnAiResults.subcall1 = null
  → SC1d无法成文(L2586-2647 early return)
  → SC1b/SC1c继续运行·但叙事_recentSZJ缺失
  → SC2追踪诏令反馈空(L1388假设p1存在)
```

关键失败链路:
1. SC1 → SC1d (硬依赖)
   若SC1失败·SC1d rescue需被改进
   成本: 失去文言实录·但shizhengji_basis基础仍可用

2. SC1 → SC_audit (元数据依赖)
   SC_audit需char_updates/fiscal_adjustments做一致性检查
   若缺失·审计不触发·坏数据通过

3. SC1 → SC2 (叙事依赖)
   SC2需p1.turn_summary和edict_feedback做诏令应声
   若缺失·叙事陷入静态概括

失败影响评分: 9/10 (致命·后续6个子调用直接受损)
